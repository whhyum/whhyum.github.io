框架分析

框架背景

Spring是一个轻量级的Java
开源框架，其是为了解决企业应用开发的复杂性而创建的。框架的主要优势之一就是其分层架构，分层架构允许使用者选择使用哪一个组件，同时为
J2EE
应用程序开发提供集成的框架。Spring使用基本的JavaBean来完成以前只可能由EJB完成的事情。然而，Spring的用途不仅限于服务器端的开发。从简单性、可测试性和松耦合的角度而言，任何Java应用都可以从Spring中受益。Spring的核心是控制反转（IoC）和面向切面（AOP）。简单来说，Spring是一个分层的JavaSE/EE
full-stack(一站式) 轻量级开源框架。

Spring作为开源的中间件，独立于各种应用服务器，甚至无须应用服务器的支持，也能提供应用服务器的功能，如声明式事务、事务处理等。不仅如此，Spring还有分层的体系结构，这意味着你能选择使用它孤立的任何部分，它的架构仍然是内在稳定的。例如，你可能选择仅仅使用Spring来简单化 JDBC 的使用，或用来管理所有的业务对象

框架对比

Spring框架优点

Spring的出现，遵从了高内聚低耦合的设计原则，方便解耦，简化开发。Spring就是一个大工厂（容器），可以将所有对象创建和依赖关系维护，交给Spring管理。Spring工厂是用于生成bean，其主要
①基于AOP编程的支持，提供面向切面编程，可以方便的实现对程序进行权限拦截、运行监控等功能；②同时其声明式事务的支持使得只需要通过配置就可以完成对事务的管理，而无需手动编程；③不仅如此，方便程序的测试，其对Junit4支持，可以通过注解方便的测试Spring程序；④
方便集成各种优秀框架，Spring不排斥各种优秀的开源框架，其内部提供了对各种优秀框架（如：Struts、Hibernate、MyBatis、Quartz等）的直接支持；⑤降低JavaEE API的使用难度，Spring
对JavaEE开发中非常难用的一些API（JDBC、JavaMail、远程调用等），都提供了封装，使这些API应用难度大大降低。

框架对比

Spring虽然有种种优点，但也有其缺点：①使用了大量的反射机制，反射机制非常占用内存。②对比Spring Boot，Spring的轻量级特性又变得不那么轻量级了，在某些组件如JSP中，代码太多，缺乏一个公用的控制器，不太适合分布式场合。

前景分析

Spring框架具有很大的前景意义，其中的Spring mvc就其融入了很好的设计理念以及其背后所传达的开发思想。而Spring框架适用于各种JavaEE企业应用开发。一般来说，如果项目工程较大，需要有合理稳定的架构，那么spring是一个很好的选择。

模式分析

层次关系

执行流程

对于Spring程序的执行流程，在Spring MVC中，首先用户发送请求至前端控制器DispatcherServlet，当DispatcherServlet收到请求后就调用HandlerMapping处理器映射器。而处理器映射器根据请求url找到具体的处理器，生成处理器对象及处理器拦截器一并返回给DispatcherServlet。DispatcherServlet收到返回信息后通过HandlerAdapter处理器适配器调用处理器，接着执行处理器，即Controller层，Controller执行完成返回ModelAndView，HandlerAdapter将controller执行结果ModelAndView返回给DispatcherServlet，DispatcherServlet将ModelAndView传给ViewReslover视图解析器。ViewReslover解析后返回具体View，接着DispatcherServlet对View进行渲染视图（即将模型数据填充至视图中），最后DispatcherServlet将对应的结果响应给用户。

在Spring IOC中，这里主要是实现bean的创建，在spring中所有的资源都用Resource表示，比如创建bean常用的方式有xml或注解，这些资源代表bean的配置清单，用于创建对应的bean。接着Spring会通过BeanDefinitionReader来定义bean读取配置，再使用BeanDefinitionRegister定义注册器，它的作用就是利用BeanDefinitionReader把我们的资源加载成BeanDefinition（Bean定义）。在获取所有的bean定义后，就开始经过一系列流程把bean都创建出来，即创建对应的bean对象。生产完放到一个 singletonObjects的Map对象池（scope = singleton的情况下），以后获取单例对象就从这个Map里获取（实现全局唯一的获取）。

模块关系

Spring框架可分类为以下的七大模块：

1、Spring Core：Core封装包是框架的最基础部分，提供IOC和依赖注入特性。这里的基础概念是BeanFactory，它提供对Factory模式的经典实现来消除对程序性单例模式的需要，并真正地允许使用者从程序逻辑中分离出依赖关系和配置。

2、Spring Context：构建于Core封装包基础上的Context封装包，提供了一种框架式的对象访问方法，比如JNDI注册器。Context封装包的特性得自于Beans封装包，并添加了对国际化I18N的支持（例如资源绑定），事件传播，资源装载的方式和Context的透明创建，比如通过Servlet容器。

3、Spring DAO：DAO (Data Access Object)提供了JDBC的抽象层，它可消除冗长的JDBC编码和解析数据库厂商特有的错误代码。并且，JDBC封装包还提供了一种比编程性更好的声明性事务管理方法，不仅仅是实现了特定接口，而且对所有的POJOs都适用。

4、Spring ORM：ORM 封装包提供了常用的“对象/关系”映射APIS的集成层。其中包括JPA、JDO、Hibernate 和 iBatis。利用ORM封装包，可以混合使用所有Spring提供的特性进行“对象/关系”映射。

5、Spring AOP：Spring的 AOP 封装包提供了符合AOP Alliance规范的面向方面的编程实现，利用AOP可以定义多种功能，例如方法拦截器和切点，从逻辑上讲，从而减弱代码的功能耦合，清晰的被分离开。而且，利用source-level的元数据功能，还可以将各种行为信息合并到你的代码中。

6、Spring Web：Spring中的 Web包提供了基础的针对Web开发的集成特性，例如多方文件上传，利用Servlet listeners进行IOC容器初始化和针对Web的ApplicationContext。当与WebWork或Struts一起使用Spring时，这个包使Spring可与其他框架结合。

7、Spring Web MVC：Spring中的MVC封装包提供了Web应用的Model-View-Controller（MVC）实现。Spring的MVC框架并不是仅仅提供一种传统的实现，它提供了一种清晰的分离模型，在领域模型代码和Web Form之间。并且，还可以借助Spring框架的其他特性。

模式分析

对于Spring而言，其中蕴含了很多的设计模式，其遵从开闭原则，使得代码可以实现更简单的拓展而不是多处的修改。本次设计主要分析了工厂方法、单例模式、原型模式、代理模式、适配器模式以及观察者模式，但其实Spring中不仅仅包含这些，深究其中还可以发现责任链模式、装饰器模式、策略模式等。

工厂方法

（一）模式应用

Spring IOC容器就像是一个工厂一样，当我们需要创建一个对象的时候，只需要配置好配置文件/注解即可，完全不用考虑对象是如何被创建出来的。IOC 容器负责创建对象，将对象连接在一起，配置这些对象，并从创建中处理这些对象的整个生命周期，直到它们被完全销毁。工厂模式最重要的类就是DefaultListableBeanFactory，其通过继承ListableBeanFactory从而继承BeanFactory，类图如图2-1所示。

图2-1 创建bean过程中涉及的工厂方法类图

根据类图所示，BeanFactory的实现原理为FactoryBean接口的bean是一类叫做factory的bean。其特点是，spring会在使用getBean调用获得该bean时，会自动调用该bean的getObject方法，所以返回的不是factory这个bean，而是这个bean.getOjbect方法的返回值。

（二）源码分析

在Spring中工厂模式有很多处的应用，这里以AbstractBeanFactory为例进行说明。首先在其getBean方法中对传入的Bean name稍做处理，防止传入的Bean name名有非法字符(或则做转码)
，接着手工检测单例注册表，查找对应的bean，若缓存中存在则返回合适的缓存Bean实例，不存在则根据Bean定义判断创建的bean类型。

总的来说，工厂方法就是自己不再主动创建对象，而是让工厂来帮我们创建对象。通常工厂提供一个getObject()方法，spring会在使用getBean调用获得该bean时，会自动调用该bean的getObject方法，所以返回的不是factory这个bean，而是这个bean.getOjbect方法的返回值。时序图如图2-2所示。

（四）模式分析

对于工厂方法模式，通常由应用程序直接使用new创建新的对象，为了将对象的创建和使用相分离，采用工厂模式,即应用程序将对象的创建及初始化职责交给工厂对象。一般情况下，应用程序有自己的工厂对象来创建Bean。如果将应用程序自己的工厂对象交给Spring管理，那么Spring管理的就不是普通的Bean，而是工厂Bean。在工厂方法模式中，用户只需要关心所需产品对应工厂，无须关心创建的细节，而且加入新产品时符合开闭原则。而Spring
IOC
解决的核心问题就是将组件的创建和组装与组件的使用相分离，并且由IOC容器管理组件的生命周期。如果没有该模式，那么对象的生产方式很容易影响客户端使用，增加了系统的耦合。

（四）模式分析

由于不会每次都新创建新对象所以使用Singleton的好处在于：①减少了新生成实例的消耗：新生成实例消耗包括两方面，第一，spring会通过反射或者CGLib来生成bean实例这都是耗性能的操作，其次给对象分配内存也会涉及复杂算法；②减少jvm垃圾回收：由于不会给每个请求都新生成bean实例，所以自然回收的对象少了；③可以快速获取到bean：因为单例的获取bean操作除了第一次生成之外其余的都是从缓存里获取的所以很快。而如果单例模式没有利用这种双锁机制，那么在多线程的不同情况也容易出现创造的实例不唯一的情况。

然而单例的bean一个很大的劣势就是他不能做到线程安全。由于所有请求都共享一个bean实例，所以这个bean要是有状态的一个bean的话可能在并发场景下出现问题，而原型的bean则不会有这样问题（但也有例外，比如他被单例bean依赖），因为给每个请求都新创建实例。

原型模式

（一）模式应用

原型模式(Prototype)是指用原型实例指定创建对象的种类，并且通过拷贝这些原型，创建新的对象，其是一种创建型设计模式，允许一个对象再创建另外一个可定制的对象，无需知道如何创建的细节。其工作原理为通过将一个原型对象传给那个要发动创建的对象，这个要发动创建的对象通过请求原型对象拷贝它们自己来实施创建，即对象.clone()

在Spring中原型模式主要应用于bean的创建，与单例模式类似，其需要在创建bean时判断其为单例还是原型，若使用原型，那么就对应的使用原型模式进行创建。

（二）源码分析

  1. if (mbd.isPrototype()) {

  2. // It’s a prototype -> create a new instance.

  3. Object prototypeInstance = null;

  4. try {

  5. beforePrototypeCreation(beanName);

  6. prototypeInstance = createBean(beanName, mbd, args);

  7. }

  8. finally {

  9. afterPrototypeCreation(beanName);

  10. }

  11. beanInstance = getObjectForBeanInstance(prototypeInstance, name, beanName,
    mbd);

  12. }

（三）模式分析

原型模式主要适用于：类初始化消耗资源较多；使用new
生成一个对象需要非常繁琐的过程（数据准备访问权限等）；构造函数比较复杂；在循环体中产生大量对象等情况以简便bean对象的创建。如果没有原型模式，那么对于构造函数比较复杂且冗余度较高的对象会产生较多的不便及多余代码。

代理模式

（一）模式应用

对于代理模式而言，其中可分为静态代理和动态代理，在SpringAOP中，就使用了动态代理来实现事务的处理，这里的动态代理还分为JDK动态代理和CGLib动态代理。JDK动态代理是基于接口的方式，即代理类和目标类（真实类）都实现同一个接口；CGLib动态代理是代理类去继承目标类（真实类），然后重写其中目标类的方法啊，这样也可以保证代理类拥有目标类的同名方法。在Spring中JDK的动态代理底层是通过Java反射机制实现的，并且需要目标对象继承自一个接口才能生成它的代理类。如图2-5所示为代理模式的简要类图。

（二）源码分析

JDK动态代理

Spring中的动态代理主要使用Proxy.newProxyInstance，但其只能代理接口且不支持抽象类，代理类都需要实现InvocationHandler类，实现invoke方法。invoke方法就是调用被代理接口的所有方法时需要调用的，返回的值是被代理接口的一个实现类。

CGLib动态代理

CGLibProxy代理类的类型是CGLibproxy.RealSubject。CGLib是一个高性能的字节码生成库，它用于在运行时扩展Java类和实现接口；本质上它是通过动态的生成一个子类去覆盖所要代理的类。和JDK动态代理不一样的是不管是接口还是类它都能正常工作。Proxy.MethodInterceptor能够满足任何的拦截需求。

在源码中，Spring主要使用Enhancer来进行创建代理，Enhancer中createHelper方法来创建代理的代码对应如下：

  1. private Object createHelper() {

  2. // 校验

  3. preValidate();

  4. Object key = KEY_FACTORY.newInstance((superclass != null) ?
    superclass.getName() : null,

  5. ReflectUtils.getNames(interfaces),

  6. filter == ALL_ZERO ? null : new WeakCacheKey<CallbackFilter>(filter),

  7. callbackTypes,

  8. useFactory,

  9. interceptDuringConstruction,

  10. serialVersionUID);

  11. // 缓存机制

  12. this.currentKey = key;

  13. // 利用字节实现并创建代理类对象

  14. Object result = super.create(key);

  15. return result;

  16. }

（三）应用流程

对于JDK动态代理而言，JDK的动态代理底层是通过Java反射机制实现的，并且需要目标对象继承自一个接口才能生成它的代理类。即利用newProxyInstance通过目标对象的类加载器以及它所继承的接口来获取代理类类型，先从缓存中获取代理类，如果不存在则通过ProxyClassFactory创建(动态代理的缓存机制)，这里通过反射获取构造器，cl是代理类型其构造器的参数类型为InvocationHandler，参数传入InvocationHandler，接着创建对应的代理对象，实现创建代理对象的功能，从而实现通过代理对象来代理真实的的目标对象。

而CGLibProxy代理类的类型是CGLibproxy.RealSubject，用于在运行时扩展Java类和实现接口；本质上它是通过动态的生成一个子类去覆盖所要代理的类。其中的Proxy.MethodInterceptor能够满足任何的拦截需求。代理模式的时序图如图2-6所示。

（四）模式分析

在SpringAOP事务处理流程中，JDK和CGLib动态代理都有各自的适用场景。JDK动态代理是基于接口的方式，其需要代理类和目标类都实现同一个接口；而CGLib动态代理是代理类去继承目标类，然后重写其中目标类的方法，这样也可以保证代理类拥有目标类的同名方法。

对比这两种方法JDK中的动态代理是通过反射类Proxy以及InvocationHandler回调接口实现的，但是，JDK中所要进行动态代理的类必须要实现一个接口，也就是说只能对该类所实现接口中定义的方法进行代理，这在实际编程中具有一定的局限性，而且使用反射的效率也并不是很高。使用CGLib实现动态代理，完全不受代理类必须实现接口的限制，而且CGLib底层采用ASM字节码生成框架，使用字节码技术生成代理类，比使用Java反射效率要高。唯一需要注意的是，CGLib不能对声明为final的方法进行代理，因为CGLib原理是动态生成被代理类的子类。

总的来说，Spring对于对象的管理是有十分友善的机制的，创建代理对象一般是通过ProxyFactoryBean去创建。该类根据Spring
bean构建AOP代理并实现了定义getObject()方法的FactoryBean接口。此方法用于将需求Bean的实例返回给bean
factory。在这种情况下，它不是返回的实例，而是AOP代理。它使得代理对象和IOC容器配置之间拥有了一座桥梁。若没有这种代理模式，对于一项功能的添加，Spring需要修改很多处地方，但是使用代理模式，可以基于代理的基础上进行拓展，从而实现了开闭原则，降低了系统的耦合度。

（四）模式分析

在Spring中，其事件监听使用的是观察者模式 所有事件需要继承ApplicationEvent父类
所有的监听器需要实现ApplicationListener接口
事件发布需要通过ApplicationContext中的publisherEvent方法实现，而监听器的注册是ApplicationEventMulticaster提供的观察者模式解除了主题和具体观察者的耦合，让耦合的双方都依赖于抽象，而不是依赖具体。从而使得各自的变化都不会影响另一边的变化。

但是，使用观察者模式也存在一定缺点，其中依赖关系并未完全解除，抽象通知者依旧依赖抽象的观察者。

通过设计案例，也能发现该模式适用于一个对象的改变需要给变其它对象时，而且它不知道具体有多少个对象有待改变时，还有一个抽象某型有两个方面，当其中一个方面依赖于另一个方面，这时用观察者模式可以将这两者封装在独立的对象中使它们各自独立地改变和复用。

  24. }

  25. public void handle(Object handler) {

  26. ((HttpController)handler).doHttpHandler();

  27. }

  28. public Boolean supports(Object handler) {

  29. return (handler instanceof HttpController);

  30. }

  31. }

添加对应的责任链，当需要增加处理者时可以在此增加对应的责任链节点：

  1. public class AdapterChain {

  2. public HandlerAdapter getAdapterChain(){

  3. HttpHandlerAdapter adapter1 = new HttpHandlerAdapter();

  4. AnnotationHandlerAdapter adapter2 = new AnnotationHandlerAdapter();

  5. SimpleHandlerAdapter adapter3 = new SimpleHandlerAdapter();

  6. adapter1.setNextHandlerAdapter(adapter2);

  7. adapter2.setNextHandlerAdapter(adapter3);

  8. return adapter1;

  9. }

  10. }

对应的DispatchServlet中只需要执行该责任链即可：

  1. public class NewDispatchServlet {

  2. public static HandlerAdapter handlerAdapter;

  3. public NewDispatchServlet(){

  4. AdapterChain chain = new AdapterChain();

  6. }

  7. public void doDispatch(){

  8. // 模拟 SpringMVC 从 request 取 handler的对象

  12. getHandler(controller2);

  13. System.out.println(“——————————“);

  14. getHandler(controller3);

  15. }

  16. public void getHandler(Controller controller){

  17. // 通过适配器执行对应的controller对应方法

  18. handlerAdapter.check(controller);

  19. }

  20. }

观察者模式

对于观察者模式，其核心在于对事件、事件发布者、观察者的定义。这里测试的案例首先要先定义事件，定义事件的核心是继承ApplicationEvent，而ApplicationEvent作为父类，构造方法要求必须传入参数Object source，表示是谁发布的事件，ApplicationEvent作为父类，是可序列化的，有serialVersionUID，建议子类也加上。接着发布事件核心是注入ApplicationEventPublisher，并使用其中的publishEvent方法 发布事件（也可以使用spring上下文容器ConfigurableApplicationContext，因为它也继承了ApplicationEventPublisher）
3、监听事件 核心是实现 ApplicationListener
接口，实现onApplicationEvent方法，方法内就是监听者对事件发生的响应（注入监听者）。
